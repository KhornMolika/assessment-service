# Assessment Service (ASP) — Project Context Summary v3

> Updated: April 2026 — reflects all architectural and flow decisions

## 1. System Overview

Assessment Service is a focused microservice that handles the full lifecycle of assessments: question preparation, answer collection, and evaluation. It is embedded into other platforms via API or micro-frontend components.

### What ASP owns
- Question banks, questions, answer options
- Assessment configuration and settings
- Answer evaluation and scoring logic
- AI grading jobs and configs
- Participant records (lightweight — no accounts)
- Answer sheets and entries
- Result calculation and grade labels
- Tenant registration and API keys
- Topic taxonomy (bridge between ASP and external services)

### What external services own
- User accounts, passwords, sessions
- Who is allowed to create / edit / delete
- Role-based access within their system
- Authentication of their own users
- Sending result emails to participants
- Gradebook / transcript management

## 2. Core Architectural Decisions (NON-NEGOTIABLE)

### No users table
ASP does not manage user identity. `tenants` is the top-level owner of all content. `created_by_ref` on banks, questions, and assessments is a plain string from the `X-Actor-ID` header — audit only, never a FK.

### No sessions table
Assessment is both the configuration and the runtime event. When a new run is needed, create a new assessment. No separate sessions concept.

### Tenant-centric ownership
```
tenants → question_banks (tenant_id)
tenants → assessments   (tenant_id)
tenants → topics        (tenant_id)
```

### Locale at tenant level only
`tenants.locale` is the single locale field in the entire schema. Flows through all assessments, components, results, and API responses via request context. No override at any other level.

### No is_correct on answer_options
Correctness lives ONLY in `questions.correct_answer` (JSONB discriminated union).

### Question snapshot on every answer_entry
```ts
question_snapshot: {
  id, question_text, type_id, points,
  settings: QuestionSettings,  // full discriminated union — never flatten
  correct_answer: CorrectAnswer
}
```
Purpose: version safety, grading independence, audit consistency.
Matching type requires full settings object — settings.left[] and settings.right[] must be preserved.

### Option IDs generated by the database
Not by the frontend. `correct_answer` stores these IDs.

### Score clamping in grading engine — always
```ts
score = Math.min(Math.max(raw, 0), question_snapshot.points)
```
Applied for ALL grading paths (AUTO, AI, MANUAL) in one place.

### question.points edit blocked on ai_grading_configs mismatch
Edit is blocked until `rubric.total_points` and `max_score` are updated to match.

### Real-time state in Redis only
Current question index, timer, participant list, and live answers for real-time mode live only in Redis during the assessment. Written to DB on completion.

```
assessment:{id}:status             → WAITING | ACTIVE | ENDED
assessment:{id}:current_question   → 0, 1, 2...
assessment:{id}:participants       → set of joined participant IDs
assessment:{id}:answers:{qIndex}   → map of participantId → answer
```

## 3. Authentication — OAuth 2.0 + JWT

Two surfaces, two auth methods.

### Server-to-server (external service backend → ASP API)
OAuth 2.0 Client Credentials flow.

```
POST /oauth/token
{ grant_type: "client_credentials", client_id, client_secret, scope: "banks:read assessments:write" }

→ ASP returns: { access_token: "eyJ...", token_type: "Bearer", expires_in: 3600 }
→ client_secret never travels again
→ access token used for all API calls (1 hour, auto-refresh)
```

### Browser / micro-frontend (participant → ASP)
Scoped short-lived session JWT. Host backend requests it server-side, passes it to the browser. Browser never sees client_secret.

```
POST /api/v1/assessments/token  (server-to-server, with OAuth access token)
{ assessment_id, external_user_id?, participant_identity }

→ ASP returns scoped JWT:
{ type: "session", tenant_id, assessment_id, participant_identity, scope: "session:take", exp: now+2h }

→ micro-frontend uses this JWT for all participant API calls
```

### Tenant dashboard login
```
POST /auth/dashboard { client_id, client_secret }
→ returns dashboard JWT (8h)
→ client_secret never stored in browser
```

### ASP admin login
```
POST /auth/admin/login { email, password }
→ returns admin JWT (8h)
→ validates against asp_admins table
```

### Scopes
| Scope | Who |
|---|---|
| banks:read / banks:write | External service backend |
| questions:read / questions:write | External service backend |
| assessments:read / assessments:write | External service backend |
| results:read | External service backend |
| grading:write | External service backend |
| session:take | Participant browser only |
| admin:* | ASP admin dashboard only |

### tenants table changes for OAuth
- `api_key_hash` → renamed to `client_secret_hash`
- `client_id` added (public identifier, not secret)
- `jwt_secret` added (per-tenant, signs session JWTs — separate from client_secret)

## 4. How External Services Integrate

### API Integration
1. Register as a tenant — status PENDING until admin approves (confirm with project owner)
2. Receive `client_id` and `client_secret` — shown once
3. Exchange for OAuth access token (1 hour) to call API
4. Pass `X-Actor-ID` header for audit trail on write operations

### Micro-frontend Embedding
1. Host backend gets a scoped session JWT from ASP (server-side)
2. Passes JWT to the embedded component: `<assess-quiz token="{jwt}" />`
3. Component calls ASP API directly from browser using JWT
4. JWT scoped to one assessment, expires in 2 hours
5. CORS enforced against `tenants.allowed_origins`

### correct_answer discriminated union
```ts

  | { type: "single";   correct_option_ids: string[] }
  | { type: "multiple"; correct_option_ids: string[] }
  | { type: "boolean";  value: boolean }
  | { type: "fill";     answers: string[] }
  | { type: "ordering"; correct_order: string[] }
  | { type: "matching"; pairs: { left: string; right: string }[] }
  | { type: "essay";    value: null }
```

## 7. Grading Engine

### Pipeline
```
Response submitted
 → grading engine detects strategy from question_snapshot.type_id
 → BINARY / DEDUCTIVE / SCALED → AUTO grading
 → MANUAL_AI → skip AUTO → create ai_grading_job → worker → LLM → Zod validate
 → score clamped in engine (ALL paths)
 → answer_entry updated
 → manual override available (grading_status = MANUAL_REVISED, graded_by_ref stored)
 → answer_sheet totals recalculated
```

### Scoring Strategies
- BINARY: Full points or 0. Types: single, true_false, ordering.
- DEDUCTIVE: rawScore = (correct × ppi) - (incorrect × ppi). Types: multiple, fill, matching.
- SCALED: finalScore = selectedValue. Rating only. Always ungraded.
- MANUAL_AI: Skips AUTO. AI for short/essay. Manual only for file_upload.

### AI Guardrails
- Zod validates AI output — deviation → job FAILED
- Score clamped in grading engine before any DB write
- Prompt injection mitigation: system constraint in every prompt
- confidence stored — low confidence flags for human review
- debug_payload stored for full audit trail
- Manual override always available — final authority

## 8. Participant Identity

`assessment_settings.participant_identity` — configured once by creator.

| Value | Who | ASP behaviour | external_user_id | display_name |
|---|---|---|---|---|
| ANONYMOUS | Anyone, no identity | No validation | null | null |
| INTERNAL | Host's authenticated users | Check X-Actor-ID header → 401 if missing | header value | null |
| EXTERNAL | Anyone with link | Show display name form (required) | null | "Sophea Chan" |

Invariant: exactly one of external_user_id or display_name is set. ANONYMOUS has both null.

## 9. Topics

Topics are each tenant's own content groupings (courses, departments, modules). They are the primary workspace-wide filter mechanism across the assessment service, not only the dashboard.

- `tenant_id` FK → tenants.id
- `slug` unique per tenant: composite constraint (tenant_id, slug)
- `parent_id` self-reference for hierarchy
- `external_id` — tenant's own system ID for cross-service resolution
- `metadata` JSONB — tenant-specific display data, stored opaquely

Junction tables (assessment_topics, question_topics, bank_topics): two FK fields only, no timestamps, hard-delete safe.

Frontend topic scope:
- The topbar topic selector is visible across the standard workspace layout.
- The selected topic is synced to `?topic=<topic-id>` and browser local storage.
- Dashboard, assessments, banks, questions, results, analytics, and search should use that selected topic to filter visible content.
- Changing the topbar topic should update visible content according to the topic id without requiring route-specific topic controls.

## 11. Assessment Modes

### Self-paced
- Each participant moves at their own pace
- Has a time limit
- Participant joins via link or QR code
- No host required

### Real-time (Kahoot-style)
- Host controls question progression
- All participants see the same question at the same time
- Participants cannot answer until timer starts
- Timer auto-starts when question is revealed
- Host can skip to correct answer before timer ends
- Per-question cycle: Reveal → Timer → Answer → Correct Answer (host clicks) → Leaderboard → Next (host clicks)

## 14. Real-time Host Flow — /assessments/[id]/host

```
Lobby (WAITING state in Redis)
   → show QR code + copy link
   → show participant list as they join
   → Start button

   → show question text and answer options
   → timer auto-starts

   → host sees live response count

## 16. Frontend State Management

### Self-paced — Zustand + persist middleware (localStorage)
```ts
{
  questions: [],
  currentIndex: 0,
  responses: {},    // Record<questionId, QuestionResponse>
  status: 'idle' | 'active' | 'submitted',
  startedAt: null,  // timestamp — NOT timeRemaining
}
```
Store `startedAt`, not `timeRemaining`. On rehydration: `remaining = timeLimit - (now - startedAt)`.
Persist: questions, currentIndex, responses, startedAt. Skip status.

### Real-time — Zustand (no persistence needed)
Real-time state comes from WebSocket events, not localStorage. No persistence — if participant refreshes they rejoin the lobby.

```ts
{
  phase: 'lobby' | 'question' | 'answer_revealed' | 'leaderboard' | 'ended',
  currentQuestion: Question | null,
  timerActive: boolean,
  selectedAnswer: string | null,
  lastResult: { correct: boolean, points: number } | null,
  leaderboard: { name: string, score: number }[],
}
```

## 17. WebSocket Events

### Host emits
| Event | Payload | Effect |
|---|---|---|
| session:start | { assessment_id } | Moves all participants from lobby to first question |
| question:skip | {} | Ends timer, moves to correct answer phase |
| question:next | {} | Reveals next question |
| session:end | {} | Ends session |

### ASP emits to host
| Event | Payload |
|---|---|
| participant:joined | { display_name } |
| answer:received | { question_index, count } |
| question:ended | { correct_option, distribution } |

### ASP emits to participants
| Event | Payload |
|---|---|
| session:started | {} |
| question:revealed | { question, options } |
| timer:started | { duration_seconds } |
| question:ended | { correct_option } |
| leaderboard:update | { top3 } |
| session:ended | { final_scores } |
